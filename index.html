⛔ Why "connected" is the dangerous word

In the gateway code (voice-agent/index.js:517-522) "connected" is emitted the moment Ultravox sends call.joined, which only means the audio bridged. A voicemail box, an iOS Live Voicemail screen, or a carrier screener all "join". The gateway even emits connected first and then emits voicemail for the same call (:548), so the connect signal is structurally inflated. CtC stores whatever the gateway posts, unverified (BrainAPIController.php:1006-1008). Hannah, meanwhile, keys "Connected" by hand from the same overstated source, which is why her number and the DB number both overstate real contact.

2 Lead-to-outcome lineage & attribution model two views, joined on the purchase date

Jamie's constraint (Slack 6/11, 3:28 PM): a redial happens on one day but the lead may have been bought yesterday or five days ago, so a naive "dials today" view files activity under the wrong cohort. The data shape lets us resolve this cleanly.

✓ The key structural fact (verified live)

ai_outbound_calls is one row per dial attempt, not one row per lead. In 30 days: 9,014 rows, 9,012 distinct ping IDs, but only 5,037 distinct leads (~1.8 dials each). The attempt index lives on each row in redial_counts (NULL/0 = first dial, 1 = first redial, up to 12), and those buckets sum exactly to 9,014. So both views come from the same table, cut two different ways. No second source is needed.

4 Per-bucket action plan what is happening, why, what to change

One plan per outcome bucket: the observation, the likely cause, and the specific self-serve change. Click-by-click numbers, agents, and field values are in sections 5 and 6.

No-answer (41.9%)

  • What: 3,779 dials rang out with no pickup and no voicemail.
  • Cause: a mix of carrier screening on burned numbers (section 5) and dialing in the lowest-pickup hours (section 6: 10am-12pm ET is 8-14% pickup).
  • Change: rotate fresh numbers per state, and shift the redial window into mid-afternoon and early evening. Both are self-serve.

Voicemail (39.4%)

  • What: 3,549 dials ended in voicemail, ~40% on the busiest numbers.
  • Cause: over-dialed, shared caller IDs get screened straight to voicemail; and the agent has no pre-call answering-machine detection, so it monologues into the box (AdjustVoiceMailCallsData.php reclassifies voicemail only after the call). CODE
  • Change (self-serve): fresh numbers spread across more lines (section 5). Change (needs dev): enable Twilio AMD at dial time so the agent stops talking into recordings; that lives in the gated /trigger-call gateway.

Media-connected: dead air / screener (~46% of connected)

  • What: the stream joins, the agent talks, nobody (or a machine) is there.
  • Cause: the inline-dial agents wait for the callee to speak first, which is good, but with no AMD and default turn-taking they hang on screeners. The known dead-air loop bug.
  • Change: add an inactivityMessages ladder with a HANG_UP_SOFT exit (Ultravox, self-serve on the agent that has an Ultravox id; the inline-dial fleet needs the change in CtC's call-creation, which is dev).

Media-connected: brief 1-2 turn bail (~48% of connected)

  • What: a human says hello, hears the opener, hangs up within 2-3 turns.
  • Cause: a weak, screener-bait opener ("Hello... am I speaking with [name]?") and a rigid 4-confirmation flow that stacks friction. Confirmed live: the rendered prompt in call_data still opens this way.
  • Change (self-serve in CtC per agent): shorten the opener to under ~7 words, collapse the four yes-gates (identity, form, interest, transfer) into one, and forbid the model from apologizing or asking "did you intend to submit". Edit the Agent Prompt field on each agent.

Transferred & billable (0.32% / 0.03%)

  • What: 29 transfers, 3 billable, $248.93 in 30 days.
  • Cause: the bottom of the funnel is starved by everything above it, and separately by the lead-feed throttle that caps how many leads ever get dialed (ManageAiOutboundCalls.php:70, no_of_leads => 1). CODE
  • Change: the reach fixes (numbers, timing, script) lift transfers; the volume ceiling itself is the feed throttle, which is dev-team work.
✓ Sequencing

Do the self-serve reach fixes first (sections 5 and 6) because they need nobody. The two highest-leverage items overall, the feed throttle and pre-call AMD, are dev-team tickets, so file them in parallel rather than waiting.

Branding vs spam stacking

The fix people reach for, branding the number, does not override a spam tag, it sits next to it. "Find Quality Insurance" is the display name (Branded Calling), but the carrier analytics vendors (TNS for Verizon, First Orion for T-Mobile, Hiya for AT&T) independently score the number's behavior. An over-dialed, shared number earns a spam reputation, and the consumer then sees the brand name and a spam warning at the same time. Branding cannot subtract a reputation the dialing pattern created. Only changing the pattern (fewer dials per line, more lines, retire burned ones) lowers the spam score.

8 SMS / RCS roadmap click-wall built, connector missing: self-serve or dev?

✓ Determination: needs a dev-team handoff (Jira ticket), not self-serve CODE

Checked the repos on disk (read-only Bitbucket access, api-callstoconvert + ui-callstoconvert): there is no consumer-facing outbound SMS send path. The SMS/RCS string hits are an admin SmartNotificationController and user-registration, and the new Client([...]) instances in PostAgentAvailability.php are Guzzle HTTP clients for agent-availability pings, not Twilio messaging. The three use cases (frame for the ticket)

  • Conversion multiplier for form-fillers: consumers who filled a form but will not answer a cold call. SMS reaches them and links to the click-wall.
  • Optionality, not coercion: gives the lead a channel choice instead of forcing a phone call.
  • Reach where calls fail: the SMS step in the sequence (section 7) for leads that go unanswered after 2-3 dials.
⚠ RCS varies by region and device, audit first

RCS support depends on carrier, region, and handset. Before any rollout, audit lead geography and device mix, and ship a plain-SMS fallback for everywhere RCS is not guaranteed. Do not assume RCS delivery; treat it as an enhancement over a reliable SMS base.

9 HLeads data-flow question (for Leo) right foundation, or restructure before dashboards?

Leads enter through HLeads and the whole system is built on top of that flow. Jamie's concern: parts of it were built as a workaround and later treated as permanent infrastructure, so building dashboards on it risks chasing technical debt.

What the live data shows about the foundation

  • The spine works, but only by name, not by id. Cross-system lead ids do not match (Leadspedia leadID ≠ CtC AL- id ≠ ping GUID, all confirmed live). Every join that crosses a system boundary leans on affiliate/publisher name matching plus date, which is fragile to renames and shared-source labels like "DMS (shared)".
  • Per-lead cost is not in the reader at all. Acquisition cost (Lead_Payout) lives in an Aged-cost DB that is not granted; the replica has only transfer-level payout. So true cost-per-outcome currently has to be approximated by source-level CPL from Leadspedia.
  • The funnel math is hidden. The numbers the existing reports show come from stored procedures (getAISummarizedStats, getAICallsAttemptReport) that are not in the repo and not executable by the reader. Any dashboard must re-implement that math, so it can silently diverge from what the platform shows.

⚠ Technical-debt risk, stated plainly

If we build dashboards on the current HLeads-fed shape, we inherit three weak joints: name-based joins, missing per-lead cost, and a re-implemented (not authoritative) funnel. None blocks a v1 prototype, but each becomes expensive to unwind once dashboards depend on it. The recommendation is to get Leo's answer before hardening the data model, not after.
